> 설명: 목록, 작성 폼, 관리자 통계 카드, 테이블, 등록 폼의 시각적 밀도와 상호작용을 함께 규정한다.

> 참조: `ui/src/pages/lessons-page.tsx`, `ui/src/pages/new-lesson-page.tsx`, `ui/src/pages/admin-dashboard-page.tsx`, `ui/src/pages/admin-agents-page.tsx`, `ui/src/pages/admin-new-agent-page.tsx`

# Knowledge And Admin Workflows Blueprint

## 1. 대상 화면

- `/lessons`
- `/lessons/new`
- `/admin`
- `/admin/agents`
- `/admin/agents/new`

## 2. 공통 기조

이 화면군은 카탈로그보다 더 생산적이어야 하고, 상세보다 더 실용적이어야 한다.  
하지만 현재처럼 건조하고 평면적인 업무 화면으로 남기지 않는다.

핵심은 아래 세 가지다.

- 읽기와 쓰기의 흐름을 살린다.
- 데이터 밀도는 높이되 시각 피로는 줄인다.
- 관리자 화면도 제품의 감성 톤 안에 묶는다.

## 3. Lessons 목록

- 현재의 테이블 중심 구조는 유지 가능하지만, 첫 인상은 `팀 지식 피드`에 더 가까워야 한다.
- 상단 소개 패널은 단순 안내 박스가 아니라 글쓰기 욕구를 자극하는 시작점이어야 한다.
- 로그인 상태에 따른 CTA 차이는 더 선명해야 한다.
- 목록 표현은 두 가지 중 하이브리드가 적합하다.
  - 데스크톱에서는 표 기반
  - 모바일에서는 카드 기반
- 제목은 가장 먼저 읽히고, 작성자와 날짜는 보조 정보로 분리한다.
- 행 hover는 단순 배경 변화가 아니라 행 전체가 살짝 들리는 느낌이 적합하다.
- Lessons 화면은 블루와 골드의 균형을 기본 무드로 사용한다.

## 4. New Lesson 작성 화면

- 이 화면은 생산성 도구이자 몰입형 글쓰기 표면이어야 한다.
- 상단 히어로는 현재보다 짧고 강하게 구성한다.
- 제목 입력은 일반 폼 필드보다 더 크고, 현재 문서의 대표값처럼 느껴져야 한다.
- 마크다운 에디터는 `작성 영역`과 `미리보기 영역`의 역할이 분명해야 한다.
- 미리보기는 단순 좌우 복제 화면이 아니라 “실제 게시 후 모습”을 암시하는 패널이어야 한다.
- 저장 버튼은 글쓰기 흐름을 끝내는 주요 액션인 만큼 hover, disabled, pending 상태가 분명해야 한다.
- 에러는 폼 상단만 아니라 문제가 발생한 구획 근처에서도 드러나야 한다.
- 제목 입력과 주요 액션은 블루 중심, 보조 안내와 생산성 포인트는 골드 중심으로 정리한다.

## 5. Admin 대시보드

- 관리 대시보드는 화려한 마케팅 화면이 아니라 `정제된 운영 콘솔`이어야 한다.
- 상단 소개 패널은 차분하지만, 숫자 카드가 너무 밋밋하면 안 된다.
- Agents, Lessons, Total Downloads 카드는 각각 독립된 성격을 가져야 한다.
- 숫자는 크게, 라벨은 단단하게, 보조 변화량이 생길 자리를 미리 고려한 구조로 문서화한다.
- 최근 등록 에이전트 목록은 평면 리스트가 아니라 활동 피드와 카드 목록의 중간 형태가 적합하다.
- Admin의 핵심 무드는 블루-보라 기반으로 유지하고, 경고와 실패 상태만 별도 상태색을 사용한다.

## 6. Admin Agents 목록

- 단순 표는 유지하되 시각적 긴장감을 추가한다.
- 헤더 행은 더 또렷하게, 데이터 행은 더 얇고 빠르게 읽히게 한다.
- 이름 열은 가장 강한 시각 계층을 가져야 한다.
- 카테고리와 엔진은 칩 또는 약한 강조 텍스트로 읽기 속도를 높인다.
- 행 전체를 링크화하지 않더라도 hover 시 `관리 가능한 객체`라는 느낌이 들어야 한다.

## 7. Admin New Agent 폼

- 긴 폼은 한 덩어리 패널 하나로 두지 않는다.
- 기본 정보, 링크/분류, 요약, 상세 설명을 시각적으로 분리한다.
- 각 섹션은 같은 카드 안의 그룹이 아니라 서로 다른 작업 단계처럼 느껴져야 한다.
- slug 자동 생성, validation error, pending 상태는 시각 피드백이 즉각적이어야 한다.
- 마크다운 설명 영역은 Lessons 작성 화면보다 더 구조적이고 기술 문서 작성에 적합해야 한다.
- 저장 버튼은 페이지 스크롤 하단에서도 쉽게 인지되도록 마지막 액션 존이 분명해야 한다.
- validation 성공과 가능 상태는 그린, 주의는 골드, 실패는 핑크 레드로 분명히 분리한다.

## 8. 상태와 피드백

- disabled 상태는 흐릿하게만 두지 말고, 조작 불가라는 의미가 명확해야 한다.
- pending 상태는 버튼 내부 텍스트만 바꾸지 말고 미세한 진행 광량이나 스피너를 허용한다.
- 성공 후 이동 흐름은 다음 화면이 자연스럽게 이어지는 느낌을 주어야 한다.
- 로그인 필요 상태는 차단 메시지보다 `행동 유도형 안내`가 적합하다.

## 9. 모바일 대응

- Lessons 목록은 모바일에서 표보다 카드 전환이 우선이다.
- 긴 Admin 폼은 섹션 간 간격과 제목 고정 규칙이 중요하다.
- 작성 화면의 버튼은 하단에 충분한 안전 여백을 둔다.
- 모바일 입력 포커스 상태에서 배경 움직임은 약화해도 된다.

## 10. 구현 우선순위

1. Lessons 목록과 작성 화면의 분위기 개선
2. Admin KPI 카드와 최근 목록 깊이 강화
3. Admin Agents 표의 시각 계층 재설계
4. Admin New Agent 폼의 섹션 분리와 상태 피드백 고도화
