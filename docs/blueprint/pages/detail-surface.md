> 목적: 에이전트 상세 화면을 더 몰입감 있고 프리미엄한 정보 경험으로 재정의한다.
> 설명: 상세 히어로, 액션 버튼, 메트릭, 탭, 마크다운 렌더링, 스냅샷 패널의 비주얼과 상호작용을 규정한다.
> 적용 패턴: 상세 경험 블루프린트 패턴
> 참조: `ui/src/pages/agent-detail-page.tsx`, `ui/src/shared/lib/markdown.tsx`

# Detail Surface Blueprint

## 1. 대상 화면

- `/blueprints/:id`
- 상세 상단 히어로
- GitHub / 다운로드 액션
- Overview / Snapshot 탭
- Markdown 렌더링 영역

## 2. 목표 인상

상세 화면은 “문서 조회”가 아니라 “정제된 제품 쇼케이스”처럼 보여야 한다.  
사용자는 이 화면에서 아래 인상을 받아야 한다.

- 이 에이전트가 하나의 완성된 자산처럼 보인다.
- 필요한 행동이 즉시 보인다.
- 기술 정보가 딱딱하지 않고 세련되게 정리되어 있다.
- 긴 마크다운 콘텐츠도 읽고 싶어진다.

## 3. 상세 히어로 규칙

- 상단 히어로는 현재의 두 칸 레이아웃을 유지하되 시각적 비중을 훨씬 더 키운다.
- 좌측에는 카테고리 배지, 생성일, 제목, 요약이 있는데, 이 요소들이 카드 안에서 줄지어 있는 느낌이 아니라 `한 장의 런치 패널`처럼 보여야 한다.
- 우측 액션 영역은 단순 버튼 묶음이 아니라 `즉시 실행 영역`처럼 분리한다.
- 히어로 배경에는 선택된 카테고리 무드에 따라 아주 약한 색 분위기 차이를 허용한다.
- 예를 들어 코딩 관련 에이전트는 블루 계열 보조광, 자동화/워크플로우는 보라 계열, 문서·생산성은 골드 계열로 연출할 수 있다.

## 4. 액션 버튼 규칙

- GitHub 버튼은 블루 기반의 가장 강한 CTA다.
- 다운로드 버튼은 보조 CTA지만 존재감이 충분해야 하며, 보라 또는 중성 표면 강조를 허용한다.
- 두 버튼 모두 hover 시 색만 변하지 않고 광량이 바뀌어야 한다.
- 버튼 내부 아이콘이 생긴다면 오른쪽으로 흐르는 진행감이 느껴져야 한다.
- 클릭 시 아주 짧은 압축과 복원 모션이 필요하다.

## 5. 메트릭 영역

- Downloads, Stars, Views는 현재보다 더 독립적인 미니 카드처럼 보여야 한다.
- 숫자는 메트릭의 주인공이어야 하며, 라벨은 한 단계 뒤로 물러난다.
- 세 개 카드가 완전히 같은 톤일 필요는 없다.
- 가장 중요한 수치 하나만 그린 또는 골드의 미세한 강조 색을 가질 수 있다.

## 6. 탭 규칙

- 탭은 단순 line 탭에서 끝내지 않는다.
- 활성 탭 아래에 얇은 바만 두는 대신, 활성 항목이 부드럽게 떠오르는 느낌이 필요하다.
- 전환 시 컨텐츠가 하드컷 되지 않고 짧은 페이드와 이동을 가져야 한다.
- Snapshot은 보조 탭이지만 존재감이 너무 약하면 안 된다.

## 7. Overview 마크다운 규칙

- Markdown 렌더러는 일반 블로그 본문처럼 보이면 안 된다.
- 제목, 표, 코드 블록, Mermaid 다이어그램이 모두 동일한 One Monokai 다크 시스템 안에 있어야 한다.
- 코드 블록은 주변 패널보다 더 낮은 명도의 인셋 공간으로 처리한다.
- 표는 흰 줄 표가 아니라 어두운 레코드 매트릭스처럼 보여야 한다.
- Mermaid 블록은 이미지처럼 붕 뜨지 않고, 래퍼 패널 안에서 안정적으로 앉아 있어야 한다.
- 긴 본문은 문단 간격으로 숨 쉬게 하고, 가독성 최대 폭을 제한한다.

## 8. Snapshot 규칙

- Snapshot은 단순 키-값 정보가 아니라 `실행 환경 카드`처럼 보여야 한다.
- Core Engine과 Slug 외에 추후 필드가 늘어나도 자연스럽게 확장될 수 있도록 모듈형 레이아웃 기준을 문서화한다.
- 카드 내부는 조용하되, hover나 포커스가 있을 때 약간의 반응성을 허용한다.

## 9. 상태 처리

- 로딩 상태는 “불러오는 중입니다” 텍스트만 두지 않는다.
- 제목 라인, 요약 라인, 메트릭, 본문 패널에 서로 다른 skeleton 리듬을 준다.
- 오류 상태는 큰 경고판이 아니라 정제된 문제 알림 패널처럼 보여야 한다.
- 잘못된 ID 접근은 일반 에러보다 더 중립적인 안내 톤으로 설계한다.

## 10. 구현 우선순위

1. 상세 히어로의 레이아웃과 무드 배경 교체
2. 액션 버튼과 메트릭 미니 카드 입체감 강화
3. 탭 전환 모션 추가
4. Markdown 렌더러 표면 스타일 재정의
5. Snapshot을 시스템 카드 톤으로 재정리
